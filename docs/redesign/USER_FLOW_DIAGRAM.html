🔴 The duplication you're seeing: one customer visit → 2 bookings (#B-101, #B-102) and 2 jobs (#J-55, #J-56). From the manager's perspective that's 4 rows for Karen's single Focus-in-bay event. Invoices split across 2 jobs. Work-log times split across 2 jobs. The customer status page has to merge both to show one timeline.

Where the terminology goes fuzzy

Word Means in DB Means in UI today Problem
booking row in bookings table — a kiosk submission or a passback Not shown to users by this name OK
check-in same row in bookings table (no separate table) Manager page /app/bookings + "Checked in" section in My Work Two UI words for one table. Manager sees "Check-ins"; tech sees "Checked in"; data layer says "bookings"
job row in jobs table — actual work performed Job cards, bay board, invoices all driven from this OK — but a passback creates a second job for the same car visit
passback a booking row with passed_from_job_id set A chip/flag on the check-in in My Work Depends on two concepts (booking + job) that the manager doesn't think of as separate things
awaiting_mechanic boolean on jobs — set when tester triggers a passback Renders as "Paused — awaiting mechanic" Not a job status but acts like one — overlaps with the status enum

Two ways to collapse to one source of truth

Pick one. Both eliminate the "2 bookings + 2 jobs per visit" problem. Recommendation at the bottom.

Option A — One booking, one job
Keep the MOT job. Don't create a second booking, don't create a second job. Passback just flips the job's current handler from tester → mechanic (and back) and appends a passback record as a child row.
jobs #J-55.current_role = 'mechanic'
job_passbacks.insert (job_id=#J-55, from_role='mot_tester',
  to_role='mechanic', items=[...], note='...')

Pros: one invoice, one customer timeline, one work-log ledger, manager sees 1 row per visit.

Cons: job "belongs to" two different queues at different times — needs a current_role column and a small migration.

Option B — Drop the passback booking
Keep the MOT job. Don't create a second booking — but DO spawn a child job for the repair. Parent-child link via jobs.parent_job_id. Mechanic queue = jobs-with-no-assignee-in-my-category, not a bookings list.
jobs #J-55.awaiting_mechanic=true
jobs.insert #J-56 parent_job_id=#J-55,
  service=maintenance, status=unassigned

Pros: each mechanic job is still its own job (useful for warranty tracking on parts), no bookings duplication.

Cons: still 2 jobs per visit, but they're explicitly linked. Invoice logic needs to roll up parent + children.

✅ Recommendation — Option A. It's the literal "one source of truth": one booking per arrival, one job per vehicle visit. The passback is modelled as an event on the job, not a new entity. Manager sees one row on the Jobs list. Karen sees one status page timeline. Invoicing is trivial. The current DB is one migration away from this — rename awaiting_mechanic to current_role (enum: mot_tester | mechanic | done), add job_passbacks child table, drop insert_passback_booking() RPC, repoint the mechanic My-Work query to jobs WHERE current_role='mechanic' AND assigned_to IS NULL. Dispose of the "passback is a booking" pattern before more code grows on top of it.
